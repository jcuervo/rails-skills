# Internationalization (i18n & Localization)

Translating user-facing text and localizing dates/numbers/currency via the
`I18n` API and `config/locales/*.yml`. Setting the request's locale is a
**request-layer** concern (it lives in a `before`/`around_action`), which is why it
sits here — and it applies to **both** app types: Web localizes views and flash,
**API** localizes error messages and serialized strings off the `Accept-Language`
header. The *rendering* of localized views is `../rails-hotwire/`; choosing and
applying the locale per request is here.

## Quick Pattern

```ruby
# config/application.rb
config.i18n.available_locales = [:en, :es, :fr]
config.i18n.default_locale    = :en
config.i18n.fallbacks         = true        # missing :es key → fall back to :en (pair with available_locales;
                                            #   some apps prefer the explicit form `= [:en]`)
```
```yaml
# config/locales/en.yml
en:
  hello: "Hello, %{name}"
  articles:
    created: "Article created"
# config/locales/es.yml
es:
  hello: "Hola, %{name}"
  articles:
    created: "Artículo creado"
```
```ruby
# app/controllers/application_controller.rb — set locale per request
around_action :switch_locale

def switch_locale(&action)
  locale = params[:locale] || locale_from_header || I18n.default_locale
  I18n.with_locale(locale, &action)
end

# API-friendly: honor the client's Accept-Language (minimal parse — first known 2-letter tag;
# for q-weighted negotiation use the http_accept_language gem or Rack's accept_language)
def locale_from_header
  request.env["HTTP_ACCEPT_LANGUAGE"]&.scan(/[a-z]{2}/)&.find { |l| I18n.available_locales.map(&:to_s).include?(l) }
end
```
```ruby
t("hello", name: "Ada")          # "Hola, Ada" under :es   — controllers/models/mailers
l(Time.current, format: :short)  # localized date/time
```
```erb
<%= t(".created") %>             <%# lazy lookup → articles.created when in articles/create %>
```

## When to reach for what

- **Web app:** set locale from `params[:locale]` (or subdomain/domain/user pref),
  translate views/flash with `t`, localize dates/money with `l`/`number_to_currency`.
- **API app:** set locale from `Accept-Language`; localize **error envelopes**
  (`../rails-api/`) and any human-readable strings. There's no view layer to
  translate — the payload carries already-localized text.
- **Persisting locale in URLs:** add `default_url_options` so generated links keep the
  locale (see Deep Dive) — Web only.

## Deep Dive

- **`with_locale` over assignment:** `I18n.with_locale(l) { ... }` (via `around_action`)
  resets the locale after the request even if it raises — safer than
  `I18n.locale = l`, which leaks across requests on a reused thread.
- **Lazy lookup:** `t(".created")` in `app/views/articles/create` resolves to
  `articles.create.created`; in a controller it resolves under
  `<controller>.<action>`. Keeps keys DRY and co-located.
- **Model + validation translations:** `activerecord.attributes.article.title` names
  attributes; `activerecord.errors.models.article.attributes.title.blank` overrides
  error text. `Article.model_name.human` / `human_attribute_name` read them.
- **Interpolation + pluralization:** `%{name}` interpolates; a key with `one:`/`other:`
  pluralizes via `t("items", count: n)`.
- **URL locale (Web):** to keep `?locale=` / `/es/...` across links:
  ```ruby
  def default_url_options = { locale: I18n.locale }
  ```
  and optionally `scope "(:locale)", locale: /en|es|fr/ do ... end` in routes
  (`routing.md`).
- **Fallbacks:** `config.i18n.fallbacks = true` (with `available_locales`) serves the
  default-locale string for a missing translation instead of a `translation missing`
  marker.
- **Organizing files:** split large catalogs — `config/locales/es/articles.yml` etc.;
  load them via `config.i18n.load_path`.

## Common Pitfalls

- **`I18n.locale =` in an action** → leaks to later requests on the same thread; use
  `I18n.with_locale` / `around_action`.
- **Locale not in `available_locales`** → `Accept-Language`/param can set an
  unexpected locale; always whitelist against `available_locales`.
- **Hardcoded strings in views/serializers** → untranslatable; route every
  user-facing string through `t`.
- **Lost locale on navigation (Web)** → links drop `?locale=`; set
  `default_url_options`.
- **`translation missing` shipped to users** → enable `fallbacks` and add a CI check
  for missing keys (e.g. `i18n-tasks`).
- **Localizing in the model with side effects** → prefer passing `count`/interpolation
  to `t`; keep locale selection at the request edge.

## Verify

```bash
# Keys resolve in each locale and interpolation works:
bin/rails runner "p I18n.t('hello', name: 'Ada', locale: :es)" 2>/dev/null         # => "Hola, Ada"
bin/rails runner "p I18n.available_locales" 2>/dev/null                            # includes the locales you configured
bin/rails runner "p I18n.t('this.key.is.missing', locale: :es).include?('translation missing') == false || I18n.fallbacks" 2>/dev/null

# Per-request locale switching (Accept-Language honored). The runner checks above prove locale
# resolution; this curl proves the around_action runs without error and (grep) that :es text renders.
bin/rails server -d && sleep 3
curl -fsS -H "Accept-Language: es" localhost:3000/ -w "\nes=%{http_code}\n" | grep -qi "hola\|artículo" && echo "✓ served under :es"
kill "$(cat tmp/pids/server.pid)"
```

Then route to `../rails-hotwire/` (translating views/flash), `../rails-api/`
(localized error envelopes), and `../rails-testing/` (assert translated output per
locale).
